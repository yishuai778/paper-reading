EMNLP 2023 · Benchmark / Tool Use

API-Bank

这篇论文不是单纯在说“模型会不会调用 API”，而是在问一个更接近真实 agent 的问题： 模型能不能先判断要不要用工具、找对工具、再把多步调用串起来完成任务。

一句话 thesis

API-Bank 通过一个可执行 benchmark、一个大规模 tool-use 训练集，以及细粒度错误分析， 把 tool-augmented LLM 的能力从“格式对不对”推进到“真实任务到底能不能完成”。

Problem Framing

它到底想补什么缺口？

当时很多工作都在展示 LLM 能接工具，但缺少一个统一框架回答三个更硬的问题： 模型现阶段到底多会用工具、怎样才能把它训得更会用工具、以及它真正卡在哪里。

这篇论文最关键的转向

从“模型会不会输出一段 API 调用格式”转向“模型在一个可执行环境里，能不能完成真实的工具增强任务”。

真实任务导向

不仅测 API call 字符串是否像样，还要看执行后是否真的满足用户意图。

能力层级导向

把 tool use 拆成 call、retrieve+call、plan+retrieve+call，而不是一个总分。

训练可闭环

不仅做 benchmark，还给出训练数据和 Lynx 微调结果来验证数据集是否有用。

Ability Ladder

论文最重要的拆法：三层工具使用能力

API-Bank 最有生命力的设计，不是它的数据量，而是它把 tool use 拆成三个难度层级。 这让你能看清模型到底是不会调用、不会找工具，还是不会规划。

Call：已知 API 时，模型能不能正确调用？

这是最低层级，问题通常是格式、参数和值是否正确。这里更像 instruction following + 参数填充。

难点格式、参数、约束理解
适合测什么调用规范性、参数抽取、基本工具意识

Retrieve + Call：不知道用哪个 API 时，能不能先找对再调用？

这已经不只是格式问题，而是工具检索问题。模型必须理解需求，匹配 API 描述，再完成调用。

难点工具检索、语义匹配、参数承接
适合测什么tool retrieval、文档理解、API 选择

Plan + Retrieve + Call：多步任务里，能不能自己规划工具链？

这是最像 agent 的部分。模型要把用户目标分解成多步，决定检索顺序和调用链条，最后再整合结果。

难点规划、状态维持、多调用串联
适合测什么agent planning、workflow orchestration、复杂工具链
Data Generation

它怎么把训练数据造出来？五个 agent 分步协作

作者认为直接让一个 LLM 一口气同时满足“domain 多样性、API 真实性、能力层级、对话质量”几乎做不好， 所以把复杂标注任务拆成了五个更简单的 agent。

Step 1

先生成领域

先把任务空间切成大量真实 domain，例如 healthcare、fitness、finance 等，为后续 API 和 query 提供语义边界。

Step 2

再生成 API 候选

根据 domain 生成可能存在的 API。为了保证“像真实世界”，输入里加入了 Public APIs 示例来约束风格和命名。

Step 3

选 API 与能力，再写 query

第三个 agent 结合“选中的 API + 指定能力层级”去写 query，从而保证样本能覆盖 call / retrieve / plan 三类难度。

Step 4

模拟 API 调用和响应

第四个 agent 负责生成工具调用轨迹和执行后响应，把 query 映射成具体的 API interaction。

Step 5

最后由 tester 自动过滤

第五个 agent 负责检查数据是否满足设计原则。论文里提到它丢弃了约 35% 的样本，保证最终质量。

这一步为什么重要

这其实是在做“合成训练数据的任务分解”。如果你要自建 tool-use 数据集，这篇论文最值得借的， 不是它的 prompt 细节，而是“把复杂标注流程拆成多个职责明确的 agent”这个思路。

Results

实验告诉了我们什么？

最重要的信号不是谁分最高，而是能力一上升到规划与检索，模型性能会急剧掉下去。

读表时最重要的判断

不能把 tool use 当成单一能力。API 调用格式学会了，不代表模型会检索 API；会检索 API，也不代表模型能跨多步规划工具链。

Alpaca 的问题

它经常连 API 都不调，说明普通 instruction tuning 还不足以让模型形成稳定工具意识。

Lynx 的问题

微调后“愿意调工具”了，但会开始幻想 API 或选错 API，工具幻觉成为主障碍。

GPT-4 的问题

不是不会调，而是先没找对工具。这说明高能力模型的瓶颈更像 retrieval/planning，而非格式。

Decision Output

这篇论文对你真正有用的三件事

如果你是为了做 agent、RAG + tools、或者 API 调用训练，这篇论文最该带走的是下面这些可迁移结构。

1. 能力拆分框架

把 tool use 分成调用、检索、规划三个层级，是做评测和训练设计时非常实用的基本框架。

2. 可执行评测系统

如果 benchmark 不执行工具，只看文本输出，你往往测到的是“格式能力”，不是“任务完成能力”。

3. 错误类型化

比起一个总分，更应区分 No Call、Hallucination、Retrieval Failure、Invalid Parameters 等失效模式。

我的判断

API-Bank 到今天依然值得读，不是因为它是最终答案，而是因为它把“工具增强 LLM”这件事第一次较系统地拆清楚了。 如果你要做自建 tool-use benchmark、数据合成、API 调用训练，这篇是很好的起点。
